## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding, tight value-pair separation |
| sm | 8px | Compact element spacing inside rows and controls |
| md | 16px | Default element spacing, form gaps, cell padding |
| lg | 24px | Section padding and card interiors |
| xl | 32px | Layout gaps between summary and detail sections |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level vertical spacing |

Exceptions: 44px minimum touch target for icon-only controls such as the valuation breakdown trigger

Source notes:
- Defaulted from locked user instruction for this session.
- Supports `01-CONTEXT.md` trust-first requirement by keeping rows readable instead of overly dense.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 | 1.5 |
| Label | 14px | 600 | 1.5 |
| Heading | 20px | 600 | 1.2 |
| Display | 28px | 600 | 1.2 |

Typography rules:
- Use exactly these four sizes: 14, 16, 20, 28.
- Use exactly these two weights: 400 and 600.
- Base-currency totals use Display.
- Section titles for Accounts and Holdings use Heading.
- Row labels, field labels, and compact metadata use Label.
- Native/base values and explanatory copy use Body.

Source notes:
- Defaulted from locked user instruction for this session.
- Chosen to keep valuation-heavy screens readable and trustworthy in a finance context.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `hsl(0 0% 100%)` | App background, primary surfaces, table background, settings panel background |
| Secondary (30%) | `hsl(210 40% 96.1%)` | Cards, grouped sections, sticky summary container, muted row bands, secondary navigation surfaces |
| Accent (10%) | `hsl(221.2 83.2% 53.3%)` | Base-currency selector, active sort/focus state, valuation breakdown trigger, primary `Save base currency` action |
| Destructive | `hsl(0 84.2% 60.2%)` | Reserved only for future destructive actions; unused in Phase 1 |

Accent reserved for: base-currency selector, active sort/focus state, valuation breakdown trigger, and primary `Save base currency` action

Color rules:
- Do not use accent for every clickable element.
- Standard links, secondary buttons, and passive metadata stay neutral.
- Converted base-currency values should rely on typography and layout for emphasis, not additional semantic colors.
- No success or warning palette is required for Phase 1 because sync trust signals and valuation warning states are deferred to later phases.

Source notes:
- Color split and accent reservation list were locked by user instruction for this session.
- Neutral surfaces fit the trust-oriented financial UI described in `01-CONTEXT.md` and `PROJECT.md`.

## Layout and Interaction Contract

### Primary screen structure
- Use a single portfolio page with this top-to-bottom order:
  1. Page header with title and supporting sentence
  2. Base-currency settings card
  3. Net worth summary card
  4. Accounts section with subtotal and rows
  5. Holdings section with subtotal and rows
- Preserve separate Accounts and Holdings sections at all times.
- Do not flatten the canonical display into one mixed asset table.

### Base currency control
- Place the base-currency selector in a dedicated settings card near the top of the page, above net worth and detail sections.
- Treat it as the single app-wide reporting control for Phase 1.
- Pair the selector with one primary action: `Save base currency`.
- After save, refresh canonical portfolio data rather than recomputing values client-side.

### Net worth summary
- Show total net worth as the most visually prominent number on the page using Display typography.
- Show the selected base currency next to the total.
- Keep supporting explanation concise: this total reflects the current app-wide base currency.

### Accounts and holdings rows
- Every row must show both:
  - original/native currency value
  - converted base-currency value
- Present the converted base-currency value with stronger emphasis than the native value.
- Keep both values visible by default in the row; do not hide one behind disclosure.
- Use a two-column value cluster on desktop-sized layouts and a stacked value cluster on narrow layouts.
- Each section must end or begin with a visible subtotal in the selected base currency.

### Valuation breakdown interaction
- Provide a lightweight inline trigger per eligible row using an icon button with a 44px touch target.
- Opening the breakdown should reveal a compact popover, sheet, or inline expansion that includes:
  - original amount
  - price used
  - FX rate used
  - converted result
- Do not navigate away from the portfolio screen.
- Do not build a full audit-trail experience in Phase 1.
- Only one breakdown needs to be open at a time.

### Loading and missing-data behavior
- Since Phase 2 source ingestion is not built yet, Phase 1 should assume seed or fixture data may be used.
- Empty state messaging must acknowledge that no portfolio data is available yet and that real source connections arrive in the next phase.
- Avoid implying that refresh/sync is already available in Phase 1.

Source notes:
- Section separation, app-wide base currency, visible native plus base values, and lightweight valuation transparency come from `01-CONTEXT.md` decisions D-01 through D-06.
- Backend-owned recalculation behavior comes from `CLAUDE.md` and `01-RESEARCH.md`.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Save base currency |
| Empty state heading | No portfolio data yet |
| Empty state body | Portfolio rows will appear once seed data is added in this phase or a source connection arrives in the next phase. |
| Error state | Unable to load portfolio valuations. Refresh the page or verify your pricing inputs and base-currency setting. |
| Destructive confirmation | none in Phase 1 |

Copy rules:
- Use direct, neutral language.
- Favor trust-building wording over promotional wording.
- Refer to “base currency” consistently; do not alternate with “reporting currency” in UI labels.
- Avoid promising live sync, alerts, or automatic fixes in this phase.

Source notes:
- Defaulted from locked user instruction for this session.
- Wording aligns with the current phase boundary in `ROADMAP.md`.

## Implementation Defaults for Planner and Executor

- Treat this UI spec as the visual source of truth even though no frontend files exist yet.
- When the frontend scaffold is created, initialize shadcn with standard React + Vite defaults rather than inventing a bespoke preset.

- Keep canonical finance values backend-supplied and render-only on the client.
